# Wintergate 서비스 소개

Wintergate는 **소규모 MSA 환경을 위한 Go 기반 비즈니스 게이트웨이**입니다.

서비스를 개발하다 보면 각 서버마다 **Request ID 부여, 인증/인가 처리, 공통 헤더 추가, 접근 권한 검증, 라우팅 정책** 같은 로직을 반복해서 구현해야 합니다. 특히 MSA 환경에서는 서비스가 늘어날수록 동일한 보안/추적/정책 코드가 각 서비스에 중복되고, 이를 관리하는 비용도 커집니다.

Wintergate는 이러한 반복 작업을 각 서비스 내부에서 직접 구현하지 않고, **서비스 앞단의 게이트웨이에서 공통 처리**하도록 만든 경량 게이트웨이입니다. Spring 서비스 기준으로는 애노테이션 기반 SDK를 통해 라우팅 정보, 인증 필요 여부, 요구 역할, 접근 정책 등을 등록하고, Wintergate는 이 정보를 바탕으로 요청을 동적으로 처리합니다.

즉, 개발자는 매번 게이트웨이 필터 코드를 직접 작성하는 대신, 서비스 코드에 필요한 정책만 선언하면 됩니다.

---

# SCG와의 차이

Wintergate는 Spring Cloud Gateway를 완전히 대체하기 위한 도구라기보다는, **목적이 다른 경량 비즈니스 게이트웨이**입니다.

Spring Cloud Gateway는 필터를 직접 작성하고 세밀하게 조작할 수 있다는 장점이 있습니다. 대규모 시스템에서 복잡한 라우팅, 서비스 디스커버리, 동적 설정, 다양한 필터 체인을 구성하기에 적합합니다.

반면 Wintergate는 실제 서비스 개발에서 자주 반복되는 패턴에 집중합니다.

- Request ID 부여
- 인증/인가 처리
- 역할 기반 접근 제어
- 공통 헤더 추가
- 서비스별 라우팅 정책 등록
- Spring 애노테이션 기반 설정 등록

즉, “게이트웨이를 직접 커스터마이징해서 쓰는 도구”라기보다는, **자주 필요한 게이트웨이 기능을 미리 표준화해두고 편하게 적용하는 도구**에 가깝습니다.

SCG가 대규모 확장성과 높은 자유도를 제공하는 범용 게이트웨이라면, Wintergate는 **소규모 서비스 팀이 별도의 복잡한 인프라 없이 공통 정책을 빠르게 적용하기 위한 실용적인 게이트웨이**입니다.

---

# Wintergate의 방향성

Wintergate가 목표로 하는 환경은 대규모 엔터프라이즈 MSA보다는, **최대 30~40개 정도의 서비스를 운영하는 소규모 MSA 또는 모듈형 백엔드 환경**입니다.

서비스가 너무 적다면 별도의 게이트웨이를 두는 비용이 더 클 수 있고, 반대로 수십~수백 개 규모의 대규모 시스템이라면 Kubernetes, Service Discovery, Config Server, Service Mesh 등과 결합된 더 복잡한 구조가 필요할 수 있습니다.

Wintergate는 그 중간 지점에 있는 팀을 대상으로 합니다.

> 서비스는 여러 개로 나뉘었지만, 아직 대규모 인프라를 구성할 정도는 아니고,
>
>
> 그렇다고 각 서비스마다 인증/인가, Request ID, 공통 정책을 반복 구현하기는 싫은 상황
>

이런 환경에서 Wintergate는 하나의 경량 게이트웨이만으로 공통 정책을 중앙화하고, Spring 애노테이션 기반 등록 방식을 통해 서비스별 설정을 간단하게 관리할 수 있도록 합니다.

---

# 장점

Wintergate의 가장 큰 장점은 **반복되는 게이트웨이 로직을 코드 작성이 아닌 선언 기반으로 처리할 수 있다는 점**입니다.

일반적으로 게이트웨이에서 인증/인가, Request ID, 헤더 처리, 라우팅 정책을 적용하려면 필터를 직접 작성하거나 설정 파일을 별도로 관리해야 합니다. Wintergate는 이러한 기능을 미리 표준화하고, 각 서비스에서는 애노테이션을 통해 필요한 정책만 등록하도록 설계되었습니다.

또한 Go 기반으로 구현되어 비교적 낮은 메모리 사용량과 빠른 처리 속도를 기대할 수 있습니다. JVM 기반 게이트웨이에 비해 가볍게 실행할 수 있으며, 소규모 서비스 환경에서 불필요하게 무거운 인프라 구성을 줄일 수 있습니다.
